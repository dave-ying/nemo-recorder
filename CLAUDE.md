# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Nemo Recorder is a browser-based, truly lossless PCM audio recorder. It captures raw audio from the Web Audio API via `AudioWorklet` (bypassing `MediaRecorder`'s lossy Opus/Vorbis encoding), and lets the user trim/split/delete/reorder segments on a waveform before exporting to WAV or MP3. It's a vanilla JS, zero-runtime-dependency, no-build-step app — plain ES modules served as static files.

## Commands

```bash
npm run dev     # node dev-server.js — static file server at http://localhost:5173 (required: ES modules don't load over file://)
npm test        # node --test "tests/**/*.test.js" — Node's built-in test runner
npm run check   # tsc --noEmit -p jsconfig.json — type-checks the JS via JSDoc annotations (checkJs, no build output)
```

Run a single test file directly: `node --test tests/waveform-math.test.js`

There is no bundler/build step — files are served as-is. `dev-server.js` is a minimal zero-dependency static server (only exists because browsers block ES module imports from `file://`); any static server works instead.

## Architecture

### Module structure (`js/`)

All modules communicate through one shared mutable `state` object (`js/state.js`) — there is no framework, no event bus, no reactive layer. Modules import `state` directly and mutate it; DOM updates are then triggered imperatively (e.g. `drawPlaybackWaveform(ratio)`, `updateSegmentCountDisplay()`). When editing behavior, trace effects through `state` rather than expecting props/events to flow.

- **`state.js`** — the single `AppState` object (JSDoc-typed) plus layout/style constants (`WAVEFORM_STYLE` colors, segment gap/corner/shadow px, `LIVE_SECONDS`). Key field groups: recording-related (`audioContext`, `mediaStream`, `recordedChunks`, `recordedTotalSamples`, `isRecording`), playback (`recordedBuffer`, `playbackOffset`, `isPlaying`), segment editing (`originalBuffer`, `segments`, `bufferEpoch` — incremented on PCM-mutating ops so undo can skip rebuild for PCM-neutral edits like split, `selectedSegmentIndex`, `hoverSegmentIndex`, `clipboardSegment` — last segment copied via Copy, context menu or Ctrl+C), effects (`effectsBuffer` — processed full-length parallel of `originalBuffer`, `loudness.enabled/targetLufs/truePeakDbtp`, `denoise.enabled/processing`, `effectScope` + per-segment `segment.fxOff` + the `PER_SEGMENT_EFFECTS` registry/color identities — see `effects.js`), live preview (`pendingTakeBuffer`, `previewOffset`, `isPreviewing`, `recordModalContext`), drag state (`draggingPlayhead`, `draggingSegmentIndex`, `pendingSegmentDrag`, `_segmentDragSnapshot` typed as `SegmentDragSnapshot|null`), audio-tool settings (`trimSilence`), and history-independents (`exportSettings`, `settings`, `micCapabilities`, `micLabel`). Start here to understand what fields exist before touching any feature.
- **`dom.js`** — one-time `getElementById` lookups into a typed `el` object, plus the three canvas 2D contexts (`liveCtx`, `waveCtx`, `reviewCtx`). Element IDs are static in `index.html`; add new DOM refs here rather than querying ad hoc.
- **`main.js`** — wires all DOM event listeners (buttons, keyboard shortcuts, drag/resize handlers) to functions exported by the other modules. This is the composition root; it contains almost no logic of its own (segment navigation, drag handling, and seek logic live in `editing.js`). Look here to find *what triggers what*, then follow into the relevant module for *how*.
- **`audio.js`** — mic connection/capabilities detection, `AudioWorklet` setup (worklet processor source is an inline string, loaded via a `data:` URL — no separate worklet file), recording start/stop, live ring-buffer used for the recording-view waveform. Channel detection does NOT trust `getCapabilities().channelCount.max` (it reports 2 for many mono-only mics); instead `probeStereoSupport()` actively asks the device for `channelCount: { exact: 2 }` in a throwaway `getUserMedia` — a mono-only device rejects it with `OverconstrainedError`, confirming mono. The probe result feeds `deriveMicCapabilities()` in `mic-detect.js`, which decides the offered option lists + preselected defaults.
- **`mic-detect.js`** — pure, DOM/state-free decision logic (`deriveMicCapabilities`) that maps a track's `getSettings()`/`getCapabilities()` plus the stereo-probe result into `{ supportedRates, supportedChannels, supportedBitDepths }` and the preselected `{ sampleRate, channels, bitDepth }` defaults. Native sample rate comes from `getSettings().sampleRate` (capability range is a last-resort fallback); stereo is offered only when actively confirmed (probe `true`) or already being delivered (native `channelCount >= 2`), never from the unreliable capability max. Kept pure so it's unit-testable in Node (`tests/mic-detect.test.js`).
- **`upload.js`** — lets the user import an existing audio file or a video file's audio track (from the empty-state upload button or the append menu), via `<input type="file">` + `AudioContext.decodeAudioData`. Rejects oversized files (>500 MB) before decoding, since `decodeAudioData` inflates to raw PCM in RAM. On decode failure, maps known-undecodable extensions (WMA/AMR/MIDI/etc.) or browser-incompatible families (OGG/Opus/WebM on WebKit, AIFF/CAF elsewhere) to a format-specific error message via `unsupportedFormatError()` in `utils.js`. Reuses `editing.js`'s `loadBufferAsRecording()` or `appendBufferToRecording()` to land in the same playback-ready state a mic capture would.
- **`waveform.js`** — canvas rendering for the playback waveform: segment "cards" with gaps, peak computation from the recorded buffer, playhead caret positioning, per-segment effect chips (drawn top-right of each card in per-segment scope — filled in the effect's identity color when on, hollow when off, with the whole card dimmed when it receives no per-segment effects; `getSegmentChipAtClientPoint` hit-tests them for click-to-toggle, `chipLayout` is the shared pure geometry), segment hover/selection (including the selection/delete pulse animation driven by `state.isHoveringTrash`, now triggered by hovering the toolbar delete button or the context menu's Delete item rather than an on-canvas control), append button positioning. Per-frame rendering is optimized via three pre-rendered offscreen layers: a base layer (card backgrounds with shadows, keyed on geometry + drag state), a played layer (waveform in played color, clipped per card), and an unplayed layer (waveform in unplayed color). Per playback/drag frame, these are blitted via `drawImage` (2-3 calls) instead of re-rasterizing the waveform path per segment. Layout reads are cached (`_cachedCanvasRect`/`_cachedViewRect`/`_cachedStageRect`/`_cachedLineOffsetTop`/`_cachedPathHeight`) and invalidated via `invalidateRectCache()` on resize to prevent forced reflows. Animation orchestration (`animateSegmentDelete`/`animateSegmentRestore`/`captureSegmentBitmap`) calls into `segment-anim.js` for canvas-level rendering. Also owns the segment-reorder drag animation rAF loop (`ensureDragAnimRunning`/`drawDragFrame`): while active, it renders the live arrangement (non-dragged segments easing toward would-be positions, a floating dragged card following the pointer with lift, scale, an opaque background, and deep shadow easing in as it's picked up, plus a neutral recessed drop-zone placeholder — deliberately styled to recede rather than match the floating card's accent color, so the two don't read as duplicates at small drag distances) and a post-release settle phase; the floating/settling card draws on a dedicated `#dragOverlayCanvas` (see `.drag-overlay-canvas` CSS — extends 28px on all four sides of the waveform and sits at z-index 3, above the ruler) so its lift + glow aren't clipped at the main canvas's top edge; `drawPlaybackWaveform` bails out while this loop or a delete/restore animation is running so the canvas frame is never clobbered.
- **`segment-anim.js`** — pure canvas rendering for delete/restore animations: shatter tiles, slide cards, snapshot capture, and animation frame drawing. Exports `drawDeleteAnimFrame`, `buildShatterTiles`, `drawShatterTiles`, `buildSlide`, `drawSlideCard`, `renderCardSnapshot`, `prepareCanvasForAnim`, `captureCanvasRegionForIndex`. Imports only from `state.js`, `dom.js`, and `waveform-math.js` — no circular dependencies.
- **`waveform-math.js`** — pure, DOM-free geometry functions (`computeSegmentBoundsPure`, `audioRatioToVisualRatio`, `visualRatioToAudioRatio`) that map between audio-time ratios and on-screen pixel ratios, accounting for the gaps drawn between segment cards. Also provides shared waveform path-building (`buildWaveformPath`, `buildOneCardPath`), peak computation (`computePeaksForRange`), and reorder helpers (`computeReorderArrangement`, `computeArrangementBounds`) that compute the would-be segment order and per-original-index target bounds during a live drag. Kept pure specifically so it's unit-testable in Node without a DOM — this is where segment/gap math bugs should be fixed, and where their tests live.
- **`editing.js`** — segment operations: split at playhead, delete segment (by index or at playhead), drag-to-reorder segment (with a live slide animation — the snapshot carries per-segment animated bounds, local waveform paths, and a settle phase that eases the floating card into its final slot), `rebuildPlaybackBuffer()` which reconstitutes `state.recordedBuffer` from the effects source (`getSourceBuffer()` — the processed parallel buffer while effects are on, else `state.originalBuffer`) + `state.segments` whenever segments change, and `loadBufferAsRecording()` which installs a brand-new full-length `AudioBuffer` (from either a finished mic capture or an uploaded file) as the current recording and runs it through the effects pipeline when effects are enabled. Edits mutate `state.originalBuffer` (paste/append/duplicate extend it; `pushHistory()` in history.js truncates the tail once undo/redo snapshots no longer reference it) and the `segments` array. All raw-growing ops funnel through `commitAppendedAudio()`: with effects off it uses the single-pass `concatAndRebuild()` helper (builds the new originalBuffer and recordedBuffer at once, halving full-buffer copies); with effects on it builds the raw concat only and awaits the effects pipeline (`requestEffectsSync({type:'append', oldLen})`), which rebuilds/redraws on commit. `state.bufferEpoch` tags snapshots so undo of PCM-neutral edits (splits) skips the rebuild. Also exports segment navigation helpers (`jumpToSegmentStart`, `jumpToSegmentEnd`), segment-reorder drag handlers (`beginSegmentReorderDrag`, `applySegmentReorderDrag`, `finishSegmentReorderDrag`, `cancelSegmentReorderDrag`), and the shared drag-seek helper (`seekFromClientX`) extracted from main.js. Copy/paste/duplicate live here too: `copySegmentByIndex()` snapshots a segment's raw PCM into `state.clipboardSegment` (raw by design — pasted audio flows through the same effects uniformly); `pasteSegmentAfterIndex()` and `duplicateSegmentByIndex()` both append cloned audio onto `originalBuffer` and splice a new segment in via the shared `insertClonedAudioAfter()`/`commitAppendedAudio()` helpers (the clip is resampled when its sample rate differs from the target); `pasteInsertAtPlayhead()` is the secondary paste mode that inserts directly at the playhead, splitting the segment there first if the playhead sits strictly mid-segment (`isPlayheadMidSegment()` reports that same condition to the context menu, so it can hide the option otherwise). All of the newly-created/pasted/duplicated segments get auto-selected afterward for visual confirmation.
- **`effects.js`** — the persistent audio-effect chain (**denoise → loudness**). Loudness normalization and noise removal are non-destructive toggles, not one-time ops: `state.originalBuffer` stays raw, and the module maintains `state.effectsBuffer`, a processed full-length parallel (every effect is length-preserving, so segment ranges index into both). **Effect scope** (`state.effectScope`, `'all'` default vs `'segment'`): per-segmentable effects (currently just noise; loudness is always whole-recording and ignores scope) can be turned on/off per segment. Each segment carries `fxOff` (array of effect keys opted out; see `PER_SEGMENT_EFFECTS` + `segmentEffectOn`/`enabledPerSegmentEffects` in state.js). In per-segment scope the noise stage's output is a **composite** (`buildDenoiseComposite`): denoised samples where a segment is on, raw where it opted out — still full-length so the `getSourceBuffer()` parity invariant holds. `computeEffectsFingerprint` includes `effectScope` + a `perSegmentNoiseSignature()` so a chip toggle/scope switch recomposites (a cheap `light` sync, no re-denoise); reordering segments with unchanged opt-outs does not. `setEffectScope()` flips scope; the per-segment toggles live in editing.js (`toggleSegmentEffect`, `setAllSegmentsEffects`) and are undoable (they travel on segments through history). `getSourceBuffer()` is the read point for playback/waveform/export (falls back to raw on length mismatch, i.e. mid-mutation). Enabled effects apply to ALL audio including future additions: append/paste/duplicate/load flows call `requestEffectsSync()`, which funnels through a single drain queue (hints: `full`, `append` — denoise only the new region incrementally, `light` — loudness stage only; merged strongest-first). Denoise results are cached per raw buffer; loudness re-runs synchronously whenever its fingerprint (buffer + settings) changes. `applyEffectsRemap(entries)` keeps caches valid across trim-silence compactions without re-denoising; `resetEffectsCaches()` runs on recording replace/teardown. Runs abort if `originalBuffer` is swapped mid-await (queued hints guarantee convergence). Effects live OUTSIDE undo history (toggling off restores raw exactly). On a processing failure the denoise effect auto-disables with a toast. Statically imports only DOM-free modules (state, loudness-normalize, rnnoise, trim-silence) so it stays Node-testable; DOM deps are lazy-loaded like trim-silence.js.
- **`history.js`** — undo/redo via snapshot stacks: `pushHistory()` saves current `segments` + `playbackOffset` + `bufferEpoch` to the undo stack and clears the redo stack, then reclaims any `originalBuffer` tail samples that only the discarded redo snapshots still referenced (one slice copy, only when reclaimable — and SKIPPED while effects are active, since effects.js keeps processed caches parallel to the full raw buffer); `popUndo()` / `popRedo()` swap between the stacks with a 50-entry cap. `resetHistory()` clears both stacks (called when a new recording/upload replaces the current audio). Imported by `main.js`, `editing.js`, `audio.js`, and `waveform.js`. Snapshots normally share the append-only buffer lineage (old segment ranges stay valid in the grown buffer, so no PCM is ever copied for undo). Trim-silence, the only PCM-*replacing* op left, instead calls `pushHistory(true)`, which pins the pre-op `originalBuffer` by reference in the snapshot; `applyHistorySnapshot()` in editing.js swaps it back on restore (and resyncs the effects pipeline), and `popUndo()`/`popRedo()` symmetrically pin the being-swapped-out buffer into the opposite stack so redo works across the boundary too. Never call plain `pushHistory()` before replacing `state.originalBuffer` — undo would restore stale segment ranges into unrelated PCM.
- **`playback.js`** — `AudioBufferSourceNode`-based playback (start/pause/seek), driven by `requestAnimationFrame` for waveform/time sync. Uses `isPlaybackActive()` to prevent the double-start race and `playbackStarting` flag (mirroring the record-modal's `previewStarting` pattern). Source nodes are tagged with identity in `onended` so stale-end events don't reset a fresh playback.
- **`scrub.js`** — held-arrow-key scrubbing with acceleration (ramps from `SCRUB_MIN_SPEED` to `SCRUB_MAX_SPEED` the longer a key is held). Stops on window blur and visibilitychange; self-terminates when no movement occurs.
- **`export.js`** — export modal UI + lazily-created Web Workers for encoding (one worker per format, created on first use). Talks to `worker-code.js` for the actual worker source. Snapshots recordedBuffer metadata at export-start so mutations during encoding don't crash the UI. For MP3, resamples to 48 kHz when the source exceeds lamejs's rate limit. Object URLs are revoked after a generous delay; workers are terminated on error so the next attempt respawns.
- **`worker-code.js`** — WAV and MP3 encoder source, kept as **template-literal strings** (not real worker files) so they can be spun up from Blob URLs and also imported directly into Node tests via `vm.runInContext`. This file is intentionally DOM-free — don't add browser-only APIs to the worker source strings. The MP3 worker downmixes >2ch input to stereo (lamejs only supports mono/stereo).
- **`record-modal.js`** — modal controller for the recording modal. Handles multiple states (`disconnected`, `ready`, `recording`, `review`), mic connection/disconnection, start/stop recording, review waveform rendering with seekable+draggable playhead (per-frame work is two `drawImage` blits against pre-rendered static ruler + unplayed layers and a pre-rendered played layer, so ruler ticks, labels, and per-column peak fills don't redraw every frame), preview playback of the pending take (deferred-seek: only restarts audio on pointerup so dragging the seek head doesn't churn audio), and the "Add to Editor" / "Retake" decision flow.
- **`help-modal.js`** — modal controller for the Help modal (opened via the "?" button in the header). Tab-switches between a Keyboard Shortcuts panel and a static Changelog panel; no state beyond which tab is active.
- **`context-menu.js`** — right-click segment context menu (Copy/Paste/Paste-at-playhead/Duplicate/Delete). `openSegmentContextMenu()` positions it at the cursor (clamped to the viewport) and toggles each item's visibility/disabled state (e.g. "Paste at playhead" is hidden unless `editing.js`'s `isPlayheadMidSegment()` is true); `initSegmentContextMenu()` wires the item clicks plus outside-click/Escape/resize/blur dismissal. "Paste at playhead" is the one item that ignores which segment was right-clicked — it always targets the playhead, matching the `Ctrl+Shift+V` shortcut in `main.js`.
- **`ui.js`** — small generic DOM helpers: toasts, empty-state display (`updateEmptyState`), transport button enable/disable (`setTransportDisabled` — also owns the three audio-tool buttons), quality-option rendering, and `attachToolbarPopover(button, popover)` (toggle + outside-click/Escape/blur/resize dismissal) for the audio-tool settings popovers.
- **`utils.js`** — pure formatting helpers (`formatTime`, `formatSize`).
- **`trim-silence.js`** — auto-trim silence action: sliding-window RMS detection (configurable threshold in dBFS and min silence duration in ms) over `state.originalBuffer`, compacts the buffer, and remaps `state.segments` so each segment's `{start, end}` points to the new sample indices (fully-silent segments are dropped). A window is silent only when EVERY channel's window RMS is below threshold (per-channel max), so stereo content living on one channel is never trimmed away. Pure DSP (`detectSilenceRegions`, `remapSegments`, `buildCompactedChannels`) is exported for unit testing; `applyTrimSilence()` uses dynamic imports for DOM-touching deps so the module stays Node-testable, and calls `effects.js`'s `applyEffectsRemap(entries)` after the swap so the effects pipeline compacts its caches with the same entries instead of re-denoising. The toolbar button + settings popover are static markup in `index.html`, wired in `main.js`.
- **`loudness-normalize.js`** — ITU-R BS.1770-4 loudness DSP (pure, no DOM/state imports): K-weighting filter (high-shelf ~1681 Hz + RLB high-pass ~38 Hz) implemented as direct IIR, integrated loudness over 400ms windows with 75% overlap and -70 LUFS absolute / -10 LU relative gating, and a true-peak limiter at the configured ceiling. True peak uses 4x windowed-sinc polyphase oversampling (not linear interpolation, which underestimates inter-sample peaks); an L1-norm bound per kernel lets it skip regions that provably can't beat the running peak. Exports `applyKWeighting`, `measureIntegratedLufs`, `findTruePeak`, `createNormalizedBuffer` (buffer creation is injectable, fully Node-testable). The persistent-effect behavior (toggle, caching, toasts, auto-apply to new audio) lives in `effects.js`; the popover toggle + settings are static markup, wired in `main.js`.
- **`rnnoise.js`** + **`rnnoise-worker.js`** — RNNoise background-noise removal. `rnnoise.js` runs on the main thread (browser-only): lazy-spawns a module-type `Worker` pointing at `rnnoise-worker.js`, manages a request-id map for in-flight calls, and exports `denoiseChannel()` — one channel in (any rate), resampled to/from 48 kHz via `OfflineAudioContext` (not linear interpolation), denoised, resampled back. It is deliberately free of state/DOM imports; the persistent-effect orchestration (channel loops, caching, busy UI, edit-mid-flight guards) lives in `effects.js`. `rnnoise-worker.js` is a real ES-module worker file (not a template-literal string like the export workers) that imports the vendored Jitsi `rnnoise-wasm` glue directly. Critically, it scales samples by 32768 before processing and back after — the RNNoise model expects 16-bit PCM magnitudes; feeding [-1, 1] floats makes suppression a no-op. `processSamples()` takes an injectable module and the worker wiring is `typeof self`-guarded, so the Node tests can run the real vendored WASM.

### Vendored dependencies

`vendor/lame.min.js` is a vendored copy of lamejs (MIT) for client-side MP3 encoding, loaded via `importScripts` inside the MP3 worker (resolved to an absolute URL since Blob-URL workers can't resolve relative `importScripts` paths). `vendor/rnnoise/` is the Jitsi `rnnoise-wasm` build (Apache-2.0) for client-side neural noise suppression, loaded as an ES module via the dedicated `rnnoise-worker.js` worker file (the WASM file is resolved through the worker's own `import.meta.url`, which is why it lives in a real worker file rather than a Blob-URL string like the export workers). There are no npm runtime dependencies — `typescript` is the only devDependency, used solely for `npm run check`.

### Editor modes (single view) + Record modal

The app has a single view (the editor). Two modes are driven by state:
- **Empty** — no `state.recordedBuffer` and not recording. Shows the empty-state placeholder with Record and upload buttons. Transport is disabled.
- **Editing** — `state.recordedBuffer` is set. Full waveform rendering, segment editing (split/delete/reorder/drag/trash/scissors), playback transport.

Recording no longer happens inline — it uses a separate modal (`record-modal.js`). The modal opens on any "Record" click (or `R` key) and handles the full lifecycle internally: mic status → record → live waveform with level meter → stop → review with seekable playback → retake or add to editor. The modal manages its own state (`disconnected`, `ready`, `recording`, `review`) independently of the editor. While the modal is open, editor keyboard shortcuts are blocked by a guard in `main.js`.

`state.originalBuffer` holds the untouched capture; `state.segments` (array of `{start, end}` sample ranges into `originalBuffer`) plus `rebuildPlaybackBuffer()` in `editing.js` produce `state.recordedBuffer`, which is what's actually played back and exported.

### Appending audio

Users can append more audio onto an existing recording (the "+" button). The append flow has two entry points:
- **Append menu → Upload**: `appendUploadedFile()` (upload.js) decodes an audio file via `AudioContext.decodeAudioData`, then calls `appendBufferToRecording()` (editing.js).
- **Append menu → Record**: Opens the record modal with `context='append'`; when the user clicks "Add to Editor", `handleModalAdd()` (record-modal.js) calls `appendBufferToRecording()`.

`appendBufferToRecording()` resamples the new audio to the original's sample rate and channel count via `OfflineAudioContext`, concatenates it onto `state.originalBuffer`, pushes a new `{start, end}` segment, and rebuilds the playback buffer. Undo restores the pre-append state.

### Segment card rendering

Segments are drawn as visually separated "cards" with gaps between them (`SEGMENT_GAP_CSS_PX`), even though the underlying audio is contiguous. This creates a mapping problem — audio-time ratios and on-screen pixel ratios diverge at segment boundaries — solved by the pure functions in `waveform-math.js`. Any change to playhead positioning, drag-handles, or click-to-seek needs to go through `audioRatioToVisualRatio`/`visualRatioToAudioRatio`, not naive linear interpolation.

## Testing

Tests are plain Node `node:test` files with no test framework/runner beyond that. `tests/waveform-math.test.js` covers the pure geometry functions; `tests/core.test.js` covers `utils.js` formatters and both encoder workers (running the WAV worker's string source via `new Function`, and the MP3 worker's via `vm.runInContext` with the real vendored lamejs loaded through a mocked `importScripts`). `tests/trim-silence.test.js` and `tests/loudness-normalize.test.js` cover those modules' pure DSP (including stereo silence detection and inter-sample true peak); `tests/rnnoise.test.js` runs the REAL vendored RNNoise WASM in Node (instantiated with the `.wasm` binary injected via `wasmBinary`, no fetch) and asserts stationary noise actually gets suppressed — do not weaken it into a shape-check; the PCM-scale contract it verifies is load-bearing. `tests/effects.test.js` covers the effects pipeline's pure helpers (channel caches, sync-hint merging, fingerprint matching, `getSourceBuffer` length-parity fallback). When adding logic that can be expressed DOM-free, prefer putting it in a pure module (like `waveform-math.js`) so it stays testable the same way.
