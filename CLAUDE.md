# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Nemo Recorder is a browser-based, truly lossless PCM audio recorder. It captures raw audio from the Web Audio API via `AudioWorklet` (bypassing `MediaRecorder`'s lossy Opus/Vorbis encoding), and lets the user trim/split/delete/reorder segments on a waveform before exporting to WAV or MP3. It's a vanilla JS, zero-runtime-dependency, no-build-step app — plain ES modules served as static files.

## Commands

```bash
npm run dev     # node dev-server.js — static file server at http://localhost:5173 (required: ES modules don't load over file://)
npm test        # node --test "tests/**/*.test.js" — Node's built-in test runner
npm run check   # tsc --noEmit -p jsconfig.json — type-checks the JS via JSDoc annotations (checkJs, no build output)
```

Run a single test file directly: `node --test tests/waveform-math.test.js`

There is no bundler/build step — files are served as-is. `dev-server.js` is a minimal zero-dependency static server (only exists because browsers block ES module imports from `file://`); any static server works instead.

## Architecture

### Module structure (`js/`)

All modules communicate through one shared mutable `state` object (`js/state.js`) — there is no framework, no event bus, no reactive layer. Modules import `state` directly and mutate it; DOM updates are then triggered imperatively (e.g. `drawPlaybackWaveform(ratio)`, `updateSegmentCountDisplay()`). When editing behavior, trace effects through `state` rather than expecting props/events to flow.

- **`state.js`** — the single `AppState` object (JSDoc-typed) plus layout/style constants (`WAVEFORM_STYLE` colors, segment gap/corner/shadow px, `LIVE_SECONDS`). Key field groups: recording-related (`audioContext`, `mediaStream`, `recordedChunks`, `recordedTotalSamples`, `isRecording`), playback (`recordedBuffer`, `playbackOffset`, `isPlaying`), segment editing (`originalBuffer`, `segments`, `bufferEpoch` — incremented on PCM-mutating ops so undo can skip rebuild for PCM-neutral edits like split, `selectedSegmentIndex`, `hoverSegmentIndex`, `clipboardSegment` — last segment copied via Copy, context menu or Ctrl+C), live preview (`pendingTakeBuffer`, `previewOffset`, `isPreviewing`, `recordModalContext`), drag state (`draggingPlayhead`, `draggingSegmentIndex`, `pendingSegmentDrag`, `_segmentDragSnapshot` typed as `SegmentDragSnapshot|null`), and history-independents (`exportSettings`, `settings`, `micCapabilities`, `micLabel`). Start here to understand what fields exist before touching any feature.
- **`dom.js`** — one-time `getElementById` lookups into a typed `el` object, plus the three canvas 2D contexts (`liveCtx`, `waveCtx`, `reviewCtx`). Element IDs are static in `index.html`; add new DOM refs here rather than querying ad hoc.
- **`main.js`** — wires all DOM event listeners (buttons, keyboard shortcuts, drag/resize handlers) to functions exported by the other modules. This is the composition root; it contains almost no logic of its own (segment navigation, drag handling, and seek logic live in `editing.js`). Look here to find *what triggers what*, then follow into the relevant module for *how*.
- **`audio.js`** — mic connection/capabilities detection, `AudioWorklet` setup (worklet processor source is an inline string, loaded via a `data:` URL — no separate worklet file), recording start/stop, live ring-buffer used for the recording-view waveform.
- **`upload.js`** — lets the user import an existing audio file or a video file's audio track (from the empty-state upload button or the append menu), via `<input type="file">` + `AudioContext.decodeAudioData`. Rejects oversized files (>500 MB) before decoding, since `decodeAudioData` inflates to raw PCM in RAM. On decode failure, maps known-undecodable extensions (WMA/AMR/MIDI/etc.) or browser-incompatible families (OGG/Opus/WebM on WebKit, AIFF/CAF elsewhere) to a format-specific error message via `unsupportedFormatError()` in `utils.js`. Reuses `editing.js`'s `loadBufferAsRecording()` or `appendBufferToRecording()` to land in the same playback-ready state a mic capture would.
- **`waveform.js`** — canvas rendering for the playback waveform: segment "cards" with gaps, peak computation from the recorded buffer, playhead caret positioning, segment hover/selection (including the selection/delete pulse animation driven by `state.isHoveringTrash`, now triggered by hovering the toolbar delete button or the context menu's Delete item rather than an on-canvas control), append button positioning. Per-frame rendering is optimized via three pre-rendered offscreen layers: a base layer (card backgrounds with shadows, keyed on geometry + drag state), a played layer (waveform in played color, clipped per card), and an unplayed layer (waveform in unplayed color). Per playback/drag frame, these are blitted via `drawImage` (2-3 calls) instead of re-rasterizing the waveform path per segment. Layout reads are cached (`_cachedCanvasRect`/`_cachedViewRect`/`_cachedStageRect`/`_cachedLineOffsetTop`/`_cachedPathHeight`) and invalidated via `invalidateRectCache()` on resize to prevent forced reflows. Animation orchestration (`animateSegmentDelete`/`animateSegmentRestore`/`captureSegmentBitmap`) calls into `segment-anim.js` for canvas-level rendering. Also owns the segment-reorder drag animation rAF loop (`ensureDragAnimRunning`/`drawDragFrame`): while active, it renders the live arrangement (non-dragged segments easing toward would-be positions, a floating dragged card following the pointer with lift, scale, an opaque background, and deep shadow easing in as it's picked up, plus a neutral recessed drop-zone placeholder — deliberately styled to recede rather than match the floating card's accent color, so the two don't read as duplicates at small drag distances) and a post-release settle phase; the floating/settling card draws on a dedicated `#dragOverlayCanvas` (see `.drag-overlay-canvas` CSS — extends 28px on all four sides of the waveform and sits at z-index 3, above the ruler) so its lift + glow aren't clipped at the main canvas's top edge; `drawPlaybackWaveform` bails out while this loop or a delete/restore animation is running so the canvas frame is never clobbered.
- **`segment-anim.js`** — pure canvas rendering for delete/restore animations: shatter tiles, slide cards, snapshot capture, and animation frame drawing. Exports `drawDeleteAnimFrame`, `buildShatterTiles`, `drawShatterTiles`, `buildSlide`, `drawSlideCard`, `renderCardSnapshot`, `prepareCanvasForAnim`, `captureCanvasRegionForIndex`. Imports only from `state.js`, `dom.js`, and `waveform-math.js` — no circular dependencies.
- **`waveform-math.js`** — pure, DOM-free geometry functions (`computeSegmentBoundsPure`, `audioRatioToVisualRatio`, `visualRatioToAudioRatio`) that map between audio-time ratios and on-screen pixel ratios, accounting for the gaps drawn between segment cards. Also provides shared waveform path-building (`buildWaveformPath`, `buildOneCardPath`), peak computation (`computePeaksForRange`), and reorder helpers (`computeReorderArrangement`, `computeArrangementBounds`) that compute the would-be segment order and per-original-index target bounds during a live drag. Kept pure specifically so it's unit-testable in Node without a DOM — this is where segment/gap math bugs should be fixed, and where their tests live.
- **`editing.js`** — segment operations: split at playhead, delete segment (by index or at playhead), drag-to-reorder segment (with a live slide animation — the snapshot carries per-segment animated bounds, local waveform paths, and a settle phase that eases the floating card into its final slot), `rebuildPlaybackBuffer()` which reconstitutes `state.recordedBuffer` from `state.originalBuffer` + `state.segments` whenever segments change, and `loadBufferAsRecording()` which installs a brand-new full-length `AudioBuffer` (from either a finished mic capture or an uploaded file) as the current recording. Edits mutate `state.originalBuffer` (paste/append/duplicate extend it; `pushHistory()` in history.js truncates the tail once undo/redo snapshots no longer reference it) and the `segments` array. A single-pass `concatAndRebuild()` helper builds both the new originalBuffer and recordedBuffer at once (halving full-buffer copies). `state.bufferEpoch` tags snapshots so undo of PCM-neutral edits (splits) skips the rebuild. Also exports segment navigation helpers (`jumpToSegmentStart`, `jumpToSegmentEnd`), segment-reorder drag handlers (`beginSegmentReorderDrag`, `applySegmentReorderDrag`, `finishSegmentReorderDrag`, `cancelSegmentReorderDrag`), and the shared drag-seek helper (`seekFromClientX`) extracted from main.js. Copy/paste/duplicate live here too: `copySegmentByIndex()` snapshots a segment's raw PCM into `state.clipboardSegment`; `pasteSegmentAfterIndex()` and `duplicateSegmentByIndex()` both append cloned audio onto `originalBuffer` and splice a new segment in via the shared `insertClonedAudioAfter()`/`concatAndRebuild()` helpers (the clip is resampled when its sample rate differs from the target); `pasteInsertAtPlayhead()` is the secondary paste mode that inserts directly at the playhead, splitting the segment there first if the playhead sits strictly mid-segment (`isPlayheadMidSegment()` reports that same condition to the context menu, so it can hide the option otherwise). All of the newly-created/pasted/duplicated segments get auto-selected afterward for visual confirmation.
- **`history.js`** — undo/redo via snapshot stacks: `pushHistory()` saves current `segments` + `playbackOffset` + `bufferEpoch` to the undo stack and clears the redo stack, then reclaims any `originalBuffer` tail samples that only the discarded redo snapshots still referenced (one slice copy, only when reclaimable); `popUndo()` / `popRedo()` swap between the stacks with a 50-entry cap. `resetHistory()` clears both stacks (called when a new recording/upload replaces the current audio). Imported by `main.js`, `editing.js`, `audio.js`, and `waveform.js`.
- **`playback.js`** — `AudioBufferSourceNode`-based playback (start/pause/seek), driven by `requestAnimationFrame` for waveform/time sync. Uses `isPlaybackActive()` to prevent the double-start race and `playbackStarting` flag (mirroring the record-modal's `previewStarting` pattern). Source nodes are tagged with identity in `onended` so stale-end events don't reset a fresh playback.
- **`scrub.js`** — held-arrow-key scrubbing with acceleration (ramps from `SCRUB_MIN_SPEED` to `SCRUB_MAX_SPEED` the longer a key is held). Stops on window blur and visibilitychange; self-terminates when no movement occurs.
- **`export.js`** — export modal UI + lazily-created Web Workers for encoding (one worker per format, created on first use). Talks to `worker-code.js` for the actual worker source. Snapshots recordedBuffer metadata at export-start so mutations during encoding don't crash the UI. For MP3, resamples to 48 kHz when the source exceeds lamejs's rate limit. Object URLs are revoked after a generous delay; workers are terminated on error so the next attempt respawns.
- **`worker-code.js`** — WAV and MP3 encoder source, kept as **template-literal strings** (not real worker files) so they can be spun up from Blob URLs and also imported directly into Node tests via `vm.runInContext`. This file is intentionally DOM-free — don't add browser-only APIs to the worker source strings. The MP3 worker downmixes >2ch input to stereo (lamejs only supports mono/stereo).
- **`record-modal.js`** — modal controller for the recording modal. Handles multiple states (`disconnected`, `ready`, `recording`, `review`), mic connection/disconnection, start/stop recording, review waveform rendering with seekable+draggable playhead (per-frame work is two `drawImage` blits against pre-rendered static ruler + unplayed layers and a pre-rendered played layer, so ruler ticks, labels, and per-column peak fills don't redraw every frame), preview playback of the pending take (deferred-seek: only restarts audio on pointerup so dragging the seek head doesn't churn audio), and the "Add to Editor" / "Retake" decision flow.
- **`help-modal.js`** — modal controller for the Help modal (opened via the "?" button in the header). Tab-switches between a Keyboard Shortcuts panel and a static Changelog panel; no state beyond which tab is active.
- **`context-menu.js`** — right-click segment context menu (Copy/Paste/Paste-at-playhead/Duplicate/Delete). `openSegmentContextMenu()` positions it at the cursor (clamped to the viewport) and toggles each item's visibility/disabled state (e.g. "Paste at playhead" is hidden unless `editing.js`'s `isPlayheadMidSegment()` is true); `initSegmentContextMenu()` wires the item clicks plus outside-click/Escape/resize/blur dismissal. "Paste at playhead" is the one item that ignores which segment was right-clicked — it always targets the playhead, matching the `Ctrl+Shift+V` shortcut in `main.js`.
- **`ui.js`** — small generic DOM helpers: toasts, empty-state display (`updateEmptyState`), transport button enable/disable, quality-option rendering.
- **`utils.js`** — pure formatting helpers (`formatTime`, `formatSize`).

### Vendored dependencies

`vendor/lame.min.js` is a vendored copy of lamejs (MIT) for client-side MP3 encoding, loaded via `importScripts` inside the MP3 worker (resolved to an absolute URL since Blob-URL workers can't resolve relative `importScripts` paths). There are no npm runtime dependencies — `typescript` is the only devDependency, used solely for `npm run check`.

### Editor modes (single view) + Record modal

The app has a single view (the editor). Two modes are driven by state:
- **Empty** — no `state.recordedBuffer` and not recording. Shows the empty-state placeholder with Record and upload buttons. Transport is disabled.
- **Editing** — `state.recordedBuffer` is set. Full waveform rendering, segment editing (split/delete/reorder/drag/trash/scissors), playback transport.

Recording no longer happens inline — it uses a separate modal (`record-modal.js`). The modal opens on any "Record" click (or `R` key) and handles the full lifecycle internally: mic status → record → live waveform with level meter → stop → review with seekable playback → retake or add to editor. The modal manages its own state (`disconnected`, `ready`, `recording`, `review`) independently of the editor. While the modal is open, editor keyboard shortcuts are blocked by a guard in `main.js`.

`state.originalBuffer` holds the untouched capture; `state.segments` (array of `{start, end}` sample ranges into `originalBuffer`) plus `rebuildPlaybackBuffer()` in `editing.js` produce `state.recordedBuffer`, which is what's actually played back and exported.

### Appending audio

Users can append more audio onto an existing recording (the "+" button). The append flow has two entry points:
- **Append menu → Upload**: `appendUploadedFile()` (upload.js) decodes an audio file via `AudioContext.decodeAudioData`, then calls `appendBufferToRecording()` (editing.js).
- **Append menu → Record**: Opens the record modal with `context='append'`; when the user clicks "Add to Editor", `handleModalAdd()` (record-modal.js) calls `appendBufferToRecording()`.

`appendBufferToRecording()` resamples the new audio to the original's sample rate and channel count via `OfflineAudioContext`, concatenates it onto `state.originalBuffer`, pushes a new `{start, end}` segment, and rebuilds the playback buffer. Undo restores the pre-append state.

### Segment card rendering

Segments are drawn as visually separated "cards" with gaps between them (`SEGMENT_GAP_CSS_PX`), even though the underlying audio is contiguous. This creates a mapping problem — audio-time ratios and on-screen pixel ratios diverge at segment boundaries — solved by the pure functions in `waveform-math.js`. Any change to playhead positioning, drag-handles, or click-to-seek needs to go through `audioRatioToVisualRatio`/`visualRatioToAudioRatio`, not naive linear interpolation.

## Testing

Tests are plain Node `node:test` files with no test framework/runner beyond that. `tests/waveform-math.test.js` covers the pure geometry functions; `tests/core.test.js` covers `utils.js` formatters and both encoder workers (running the WAV worker's string source via `new Function`, and the MP3 worker's via `vm.runInContext` with the real vendored lamejs loaded through a mocked `importScripts`). When adding logic that can be expressed DOM-free, prefer putting it in a pure module (like `waveform-math.js`) so it stays testable the same way.
